## Historia

1969 => PMI
PMBoK => 1996

Desde siempre que ha existido la gestión y planificación de proyectos. En ingenieria de software se trata de una disciplina heredada de muchas otras. Al igual que cualquier otro tipo de proyecto (que no sea de software), las prácticas, metodologías y herramientas pueden ser las mismas o muy similares.

Es importante hacer una aclaración respecto a dos conceptos importantes:

1. Proyecto: lo dirige un gerente (jefe de proyecto)
2. Programa: lo dirige un director (orquestados de un programa, que puede contener muchos proyectos). Por ejemplo, un programa espacial, que involucra una serie de proyectos asociados.

Para los proyectos de TI, existe algunos métodos, por ejemplo: PRINCE2 e ITIL.

Referencia: https://en.wikipedia.org/wiki/PRINCE2
Referencia: https://en.wikipedia.org/wiki/ITIL

Ahora, otra diferencia importante se debe dar a los siguientes 2 conceptos:

1. Operación: que es un conjunto de tareas, de naturaleza repetitiva.
2. Proyecto: un esfuerzo de naturaleza única.

## Estructura de un proyecto

Un proyecto se define por varios atributos:

1. Tiene un inicio;
2. Tiene un fin;
3. Tiene un objetivo que es de carácter único, basando en las condiciones en las cuales se desarrolla;
4. Durante su existencia, se va elaborando de forma progresiva (paso a paso);
5. Se mide:
	1. Por las necesidades (los requerimientos);
	2. Por las expectativas (requerimientos implícitos: revenue, costos, venta, etc)
6. Tiene stakeholders, que se refiere a los involucrados/interesados:
	1. Puede ser algo positivo;
	2. O puede ser algo negativo;
	3. Requiere una gestión: identificarlos, manejarlos, en especial sus expectativas.
7. Requiere COMUNICACIÓN.

## Factores del éxito de un proyecto

Se puede resumir por lo siguiente:

1. Lograr el o los objetivos planteados, comunicados y acordados;
2. Haberlos logrado dentro del plazo acordado o planificado;
3. Haberlos conseguido con el presupuesto acordado, aprobado y acordado;
4. Haber conseguido los adecuados niveles de calidad (QA);

## Integración

Un proyecto puede abarcar multiples procesos, contextos, personas, interfaces, etc. En un proyecto puede haber involucrado además multiples productos. Todos estos factores hacen que gestionar un proyecto sea una tarea de integración compleja.

Para gestionar estos procesos de integración existe un marco de trabajo: <mark style="background: #FFF3A3A6;">PMBoK</mark>. Se trata de un conjunto de guías, metodologías y prácticas para la gestión de proyectos.

Referencia: https://en.wikipedia.org/wiki/Project_Management_Body_of_Knowledge

El PMBok se estructura de la siguiente manera:

1. Grupos de procesos:
	1. Initiating; 
	2. Planning;
	3. Executing;
	4. Monitoring and controlling;
	5. Closing;
2. Áreas de conocimiento:
	1. Project <mark style="background: #FFF3A3A6;">integration</mark> mgr;
	2. Project <mark style="background: #ADCCFFA6;">scope</mark> mgr;
	3. Project <mark style="background: #ABF7F7A6;">schedule</mark> mgr;
	4. Project <mark style="background: #BBFABBA6;">cost</mark> mgr;
	5. Project <mark style="background: #CACFD9A6;">quality</mark> mgr;
	6. Project <mark style="background: #CACFD9A6;">resource</mark> mgr;
	7. Project <mark style="background: #FFB86CA6;">communication</mark> mgr;
	8. Project <mark style="background: #ABF7F7A6;">risk</mark> mgr;
	9. Project <mark style="background: #ADCCFFA6;">procurement</mark> mgr;
	10. Project <mark style="background: #FFF3A3A6;">stakeholder engagement</mark>;

Estas áreas de conocimiento dependen mucho del "tipo" del proyecto y el contexto en el cual se desarrolla. 

Un concepto importante es que el ciclo de vida de un proyecto no es el mismo que el ciclo de vida de un producto. Un producto puede ser el resultado de un proyecto, pero su vida se puede extender mucho más allá. El proyecto tiene un horizonte bien definido, se podría decir que los proyectos son un concepto mucho más efímero que un producto.
